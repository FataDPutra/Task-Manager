Q: Kenapa akhirnya memilih PostgreSQL dibanding MySQL?
A: Saya butuh fitur yang lebih kaya buat tracking task, misalnya tipe data JSONB untuk menyimpan metadata filter dan dukungan full-text search yang lebih matang. Selain itu, PostgreSQL lebih ketat soal referential integrity, jadi relasi user â†” task (seperti di migration `2025_11_06_171604_create_tasks_table.php`) aman tanpa harus nambah trigger manual.

Q: Alasan utama memakai Eloquent dari Laravel apa?
A: Eloquent sudah nyatu sama cara pikir Laravel: model `Task` tinggal mendefinisikan relasi `belongsTo` dan `hasMany`, lalu service seperti `TaskService` bisa fokus ke logika bisnis. Query builder-nya juga rapi, jadi validasi di `FormRequest` plus response di trait `ApiResponse` terasa konsisten.

Q: Fungsinya Docker di proyek ini apa saja?
A: Ada tiga kontainer inti (app, nginx, db) di `backend/docker-compose.yml` dan satu kontainer frontend. Dengan Docker, saya tinggal `docker-compose up -d` dan seluruh stack (PHP 8.2 + PostgreSQL + React) nyala dengan konfigurasi identik, tanpa drama beda versi di tiap mesin tim.

Q: Kenapa backend-nya pakai Laravel, bukan Express.js?
A: Laravel sudah siap pakai untuk kebutuhan saya: ada migration, seeder, queue, middleware, sampai JWT bisa dipasang lewat `tymon/jwt-auth` tanpa banyak glue code. Kalau Express, saya harus rakit semua layer manual (routing, validation, ORM) dan waktu pengembangan pasti lebih panjang.

Q: Kenapa frontend-nya pilih React?
A: React 19 memberi saya kontrol penuh untuk bikin SPA interaktif. Komponen `TaskList`, `TaskForm`, `TaskDetail` gampang dipecah, sementara React Router memudahkan proteksi route `/tasks`. Ekosistem tooling (Hooks, Context, React Query kalau nanti dibutuhkan) sudah familiar buat tim.

Q: Kenapa tidak pakai Inertia atau stack Laravel lain?
A: Saya sengaja pisahkan frontend dan backend supaya deployment fleksibel. React butuh ekosistemnya sendiri (Tailwind, testing, future SSR). Kalau pakai Inertia, saya tetap terikat dengan lifecycle Laravel Blade sementara saya pengin tim frontend bisa iterate tanpa harus sentuh repo PHP.

Q: Bisa jelasin struktur folder backend yang sekarang?
A: Di `backend/src/app` saya pisahkan layer dengan jelas: `Http/Controllers` menangani request, `Services` isi logika seperti `AuthService` & `TaskService`, `Requests` untuk validasi, `Traits/ApiResponse` buat wrapper respons, sementara `Utils` simpan helper (logger & sanitizer). Pattern ini bantu testing karena masing-masing tanggung jawab kecil.

Q: Struktur frontend kenapa dibikin begitu?
A: Folder `components/` saya isi UI deklaratif (Login, Register, TaskList, TaskItem). `context/AuthContext.js` pegang auth global, `services/api.js` satukan seluruh call Axios biar gampang ganti base URL, lalu `hooks/` untuk helper reusable seperti pengelola form. Dengan layout ini, saat butuh fitur baru cukup tambahkan komponen dan hubungkan ke context tanpa bongkar struktur besar.

Q: State management-nya gimana?
A: Saya pakai Context API untuk auth karena kebutuhannya simpel: simpan `token` dan `user` di localStorage lalu expose lewat `AuthProvider`. Untuk data task, pendekatan yang dipakai masih state lokal di `TaskList` yang fetch via `services/api.js`. Kalau nanti kompleks, tinggal tambahin React Query atau Zustand, tapi buat scope sekarang ini sudah ringan dan gampang di-debug.

Q: Bisa share langkah step-by-step waktu bikin aplikasi ini?
A: Urutannya kira-kira gini: (1) bootstrap repo dan bikin Dockerfile + docker-compose backend/frontend, (2) setup Laravel, generate model + migration `tasks`, (3) pasang JWT dan bikin `AuthController` + `TaskController`, (4) rapikan service layer & response trait, (5) bikin React app, susun AuthContext dan halaman login/register, (6) bangun komponen task (list, form, detail, filter), (7) styling dengan Tailwind, (8) tulis README + script `create-dump.sh` & `restore-dump.sh`, terakhir (9) tes end-to-end pakai Postman & screenshot untuk dokumentasi.
